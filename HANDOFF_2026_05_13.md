## Workflow rules — critical for this codebase

These are documented in `WORKFLOW.md` but worth surfacing because they were established during the previous session and a new Mac Claude must respect them from the first message:

1. **Mac Claude does not write code, write scripts, or run shell commands.** Mac Claude reads files and writes documents (INSTRUCTIONS files, doc updates, tracker entries). Anything that runs on a machine is Claude Code's job.

2. **Mike handles git commits personally** for non-Claude-Code edits (Mac Claude's doc updates). When asking him to commit, give him a command-line-friendly single-line `git add && git commit -m "..." -m "..."` block, not a multi-step procedure.

3. **Mike has ADHD.** Per his user preferences: precise, concise replies; no preambles, no recaps, no motivational language; short declarative sentences. He has told this session multiple times to slow down, to stop flooding with text, and to ask one question at a time. When asking him a question, **ask one at a time** and use the `ask_user_input_v0` tool when there are options to choose from. Do not stack multiple questions in a single message.

4. **Mac Claude does not coin new technical terms.** Use standard terminology or describe in plain English. If a phrase sounds technical but is invented, drop it. Mike caught and corrected this multiple times in the previous session ("two-script tail" was an invented phrase that confused him).

5. **Pre-flight context is mandatory** before drafting any INSTRUCTIONS file. See `WORKFLOW.md` "INSTRUCTIONS File Format" section for the required subsections. The previous session shipped a spec with a Pre-flight that missed real drift (Edge Functions using service role keys, item C1) — the implementer caught it during execution. Do better.

6. **Sites this affects** section is mandatory at the bottom of every spec. Every file in the repo that references the concept the spec changes must be in one of three buckets: Modified, Intentionally unchanged, or Deferred.

7. **Scope expansion rule for Claude Code:** if Claude Code finds mechanical extensions of the same change (e.g., same testID in more tests than the spec listed), it extends the fix and notes it in the report. If it finds genuinely different adjacent broken code, it stops and asks Mike. Spec authors should anticipate the first case via thorough Pre-flight grep.

8. **IMPL_SHA must be captured before `simplify-and-verify.sh` runs.** That script lands a chore commit on top of HEAD; the reviewer (`run-review.sh`) must run against the implementation commit, not the chore commit. The verification template at the end of every INSTRUCTIONS file enforces this:
   ```
   IMPL_SHA="$(git rev-parse HEAD)"
   bash simplify-and-verify.sh
   bash run-review.sh "$IMPL_SHA" <path to spec>
   ```

9. **Drift detector threshold changes require their own INSTRUCTIONS file.** Mid-pipeline acceptance of an unexpected count is forbidden. The threshold table in `TRIVOLTA_TRACKER.md` (Drift Detector section) is authoritative.
